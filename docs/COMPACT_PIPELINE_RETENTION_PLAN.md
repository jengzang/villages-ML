# Compact Pipeline Retention Plan

> For implementation agents: treat this as the source of truth for compact-mode table retention until it is superseded by a newer docs file. Do not implement destructive cleanup without tests and an explicit confirmation of this table list.

**Goal:** Define the compact pipeline database shape for national-scale village-name analysis.

**Scope:** Compact mode keeps global and regional analysis outputs, plus lightweight metadata and character similarity. It drops or skips village-level derived tables that scale with the number of natural villages and only support village-detail APIs.

**Decision:** `char_similarity` is retained in compact mode.

---

## Compact Definition

Compact mode is an analysis-result database, not a full village-detail database.

It should support:

- Character frequency, tendency, significance, embeddings, and character similarity/network APIs.
- Semantic category, subcategory, and regional index APIs.
- N-gram frequency, tendency, significance, and structural pattern APIs.
- Morphology pattern frequency and regional tendency APIs.
- Region similarity and region clustering APIs.
- Spatial hotspot and regional spatial summary APIs.
- Pipeline metadata, active run selection, and query-policy metadata.

It does not need to support:

- Per-village n-gram lookup.
- Per-village semantic structure lookup.
- Per-village feature vectors.
- Per-village spatial feature/detail lookup.
- Village-detail API payloads that include ngrams, semantic structure, spatial features, and feature vectors.
- Online subset compute flows that require `village_features`.

## Splitting Logic

Use these rules when deciding whether a table belongs in compact mode.

1. **Granularity rule**

   Keep tables where one row represents a global result, region result, character, semantic category, n-gram, pattern, cluster, hotspot, or run.

   Drop tables where one row represents a natural village or a natural-village-to-feature assignment.

2. **API capability rule**

   Compact keeps APIs for regional/global analysis. Compact does not promise village-detail APIs.

3. **Final-result rule**

   Keep final analysis outputs. Drop intermediate tables if the backend does not need to query them after the pipeline completes.

4. **Size-risk rule**

   Tables scaling as `villages`, `villages x features`, or `villages x assignments` are first-class cleanup targets. Tables scaling as `characters x top_k` are acceptable if they support active APIs.

5. **Character similarity exception**

   `char_similarity` is a precomputed character-level result, not a village-level table. It is required by current backend character similarity and character network APIs, so compact mode keeps it.

## Tables To Drop Or Skip

These are compact cleanup targets because they are village-level derived tables.

- `village_ngrams`
- `village_semantic_structure`
- `village_features`
- `village_spatial_features`
- `village_cluster_assignments`

The backend currently queries these tables in village-detail, village-search, and subset-compute paths. Compact mode must either disable those routes for compact databases or make those routes return a controlled degraded response when the tables are absent.

## Raw And Preprocessed Village Tables

Compact mode keeps the raw village table but does not keep the preprocessed village table.

Keep:

- `广东省自然村`
- `全国自然村`

Drop or omit after derived outputs are built:

- `广东省自然村_预处理`
- `全国自然村_预处理`

Rationale:

- Raw data is retained for manual reruns and debugging.
- The preprocessed table is large and reproducible.
- Metadata APIs do not need the preprocessed table after backend reads move to existing small tables.

Existing compact metadata tables:

- `metadata_overview_stats`
- `region_hierarchy_stats`
- `regional_centroids`

`region_catalog` is no longer required for `/metadata/stats/*`. It may still be useful later as a richer unified region lookup, but it is optional and should not block preprocessed-table cleanup.

## National Profile Changes

Set Phase 1 in `config/pipeline.national.json` to keep compact character similarity:

```json
{
  "precompute_similarities": true,
  "top_k": 20
}
```

`top_k: 20` is the recommended national compact default because it preserves the character similarity and character network APIs while keeping the table bounded by `characters x 20`.
